真实接口配置指南
================

这份文档是给第一次接触后端项目、API、配置文件的小白看的。

你可以先记住一句话：

当前项目已经可以在 Mock 模式下完整跑通，但 Mock 是“假的外部服务”。
如果要接入真实业务，优先只改两个地方：

1. .env
   用来决定启用 mock 还是真实 Provider。

2. config/external_interfaces.yaml
   用来填写真实 URL、API Key、请求头、请求体、返回字段映射。

普通 HTTP + JSON 接口一般不用改业务代码。
只有遇到非常特殊的接口，比如签名、加密、多次请求合并、分页拉取，才需要补 Provider 适配代码。


一、先搞懂这个项目有哪些“接口”
--------------------------------

项目启动后，浏览器打开：

http://127.0.0.1:8000/docs

你会看到 FastAPI 自动生成的接口文档。

目前暴露出来的主要接口有：

1. GET /api/v1/health

作用：检查服务是否正常运行。

调用成功会返回：

{
  "status": "ok",
  "app": "smart-customer-service"
}

如果这个接口都不通，说明后端服务没有启动成功。


2. POST /api/v1/chat

作用：聊天主入口。

你发给它：

{
  "sender_id": "user_001",
  "message": "帮我查订单"
}

它会返回：

{
  "sender_id": "user_001",
  "message": "请提供订单号"
}

这里 sender_id 是“用户身份”。同一个 sender_id 才能连续对话。
比如先问“帮我查订单”，再发“10001”，系统才知道你是在补充订单号。


3. GET /api/v1/providers/status

作用：查看当前项目使用的是哪些 Provider。

Provider 可以理解成“外部能力供应商”：

- LLM Provider：大模型，比如 DeepSeek、Qwen、OpenAI、Ollama。
- Business Provider：业务系统，比如订单系统、售后系统、会员系统。
- Handoff Provider：人工客服系统，比如工单系统、客服 Webhook。
- Vector Store：知识库向量检索，比如 memory、pgvector。


4. POST /api/v1/knowledge/import

作用：知识库导入接口。

当前版本是 Mock/占位实现。
现在知识文档默认从 data/knowledge/ 目录读取 Markdown 文件。

如果以后要支持“后台上传文档、自动切分、生成向量、写入 pgvector”，就要扩展这个接口。


二、先搞懂 Mock 模式
-------------------

项目默认是 Mock 模式。

Mock 的意思是：假装有真实服务，但其实是代码里写好的模拟逻辑。

默认 .env.example 里是这样：

LLM_PROVIDER=mock
BUSINESS_PROVIDER=mock
HANDOFF_PROVIDER=mock
VECTOR_STORE=memory
TRACKER_STORE=memory

所以你不用买大模型，不用订单系统，不用数据库，也能测试：

用户：帮我查订单
客服：请提供订单号
用户：10001
客服：订单已发货，物流单号为 SF10001

但是注意：

订单号 10001 和物流单号 SF10001 是模拟数据。
人工客服工单号 MOCK-XXXXXX 也是模拟数据。
退货政策来自 data/knowledge/return_policy.md 示例文件。


三、真实运行前要准备什么
------------------------

如果你要让这个项目真正接入业务，通常需要准备：

1. 一个大模型 API

例如：

- DeepSeek
- Qwen
- OpenAI
- Ollama 本地模型

你需要知道：

- API 地址
- API Key
- 模型名称


2. 一个真实订单系统接口

你需要知道：

- 查询订单的接口地址
- 请求方式，是 GET 还是 POST
- 请求参数叫什么，比如 order_id 还是 orderNo
- 鉴权方式，比如 Bearer Token、API Key、签名
- 返回数据格式，比如 status、tracking_no 字段在哪里


3. 一个真实人工客服接口

你需要知道：

- 创建工单接口地址
- 请求参数格式
- 鉴权方式
- 返回的工单号字段叫什么


4. 可选：PostgreSQL + pgvector

如果你要用真实向量知识库，需要准备：

- PostgreSQL 数据库
- pgvector 扩展
- 向量表结构
- Embedding 模型

本项目 Docker Compose 已经准备了 PostgreSQL + pgvector 镜像。


四、配置文件在哪里
------------------

项目根目录有一个环境变量模板：

.env.example

第一次运行时复制一份：

cp .env.example .env

这个 .env 主要放“开关类配置”，比如：

- 当前用 mock 还是真实接口
- 配置文件路径
- 数据库地址
- Redis 地址
- 运行端口

真实外部接口的 URL、API Key、字段映射，建议统一放到：

config/external_interfaces.yaml

这个文件要从模板复制：

cp config/external_interfaces.example.yaml config/external_interfaces.yaml

然后在 .env 里确认：

EXTERNAL_INTERFACES_CONFIG=config/external_interfaces.yaml

不要把 .env 上传到 GitHub，因为里面可能有 API Key、数据库密码。
不要把 config/external_interfaces.yaml 上传到 GitHub，因为里面也可能有 API Key。
项目里的 .gitignore 已经忽略了这两个文件。

可以上传 GitHub 的是：

config/external_interfaces.example.yaml

它只是模板，不应该写真实密钥。


五、接入真实大模型
------------------

现在推荐使用统一外部接口配置文件来接真实大模型。

第一步，改 .env：

LLM_PROVIDER=http
EXTERNAL_INTERFACES_CONFIG=config/external_interfaces.yaml

第二步，复制模板：

cp config/external_interfaces.example.yaml config/external_interfaces.yaml

第三步，修改 config/external_interfaces.yaml 里的 llm 段：

llm:
  enabled: true
  url: "真实大模型接口地址"
  method: "POST"
  api_key: "真实 API Key"
  model: "真实模型名"
  timeout: 60
  headers:
    Authorization: "Bearer {api_key}"
    Content-Type: "application/json"
  body:
    model: "{model}"
    temperature: 0.1
    messages:
      - role: "system"
        content: "你是客服意图识别器，只返回 JSON，不要输出解释。"
      - role: "user"
        content: "{prompt}"
  content_path: "choices.0.message.content"

这里几个字段是什么意思：

- enabled：是否启用这个真实接口。
- url：真实大模型接口地址。
- api_key：你的密钥。
- model：模型名称。
- headers：请求头，{api_key} 会自动替换。
- body：请求体，{prompt} 会自动替换成系统生成的提示词。
- content_path：从接口返回 JSON 里取回答内容的位置。


1. 接入 DeepSeek

.env 示例：

LLM_PROVIDER=http
EXTERNAL_INTERFACES_CONFIG=config/external_interfaces.yaml

config/external_interfaces.yaml 的 llm 示例：

llm:
  enabled: true
  url: "https://api.deepseek.com/chat/completions"
  method: "POST"
  api_key: "你的DeepSeek密钥"
  model: "deepseek-chat"
  headers:
    Authorization: "Bearer {api_key}"
    Content-Type: "application/json"
  body:
    model: "{model}"
    temperature: 0.1
    messages:
      - role: "system"
        content: "你是客服意图识别器，只返回 JSON，不要输出解释。"
      - role: "user"
        content: "{prompt}"
  content_path: "choices.0.message.content"


2. 接入 Qwen

.env 示例：

LLM_PROVIDER=http
EXTERNAL_INTERFACES_CONFIG=config/external_interfaces.yaml

config/external_interfaces.yaml 的 llm.url 示例：

https://dashscope.aliyuncs.com/compatible-mode/v1/chat/completions

model 可以填：

qwen-plus


3. 接入 OpenAI

.env 示例：

LLM_PROVIDER=http
EXTERNAL_INTERFACES_CONFIG=config/external_interfaces.yaml

config/external_interfaces.yaml 的 llm.url 示例：

https://api.openai.com/v1/chat/completions

model 可以填：

gpt-4.1-mini


4. 接入 Ollama

Ollama 是本地模型，不一定需要 API Key。

.env 示例：

LLM_PROVIDER=http
EXTERNAL_INTERFACES_CONFIG=config/external_interfaces.yaml

Ollama 的接口格式和 OpenAI 兼容接口可能不一样。
如果你使用 Ollama OpenAI 兼容接口，llm.url 可以写：

http://localhost:11434/v1/chat/completions

model 可以写：

qwen2.5


重要说明：

现在代码里已经有 HttpLLMProvider。
普通 OpenAI 兼容接口可以通过 config/external_interfaces.yaml 配置，不需要新增 Provider。

但是大模型必须返回 CommandParser 能识别的 JSON。
例如识别“查订单”时，大模型最终内容要是：

{
  "type": "start_flow",
  "name": "query_order",
  "confidence": 0.95
}

识别“知识问答”时：

{
  "type": "knowledge_qa",
  "confidence": 0.95
}

业务代码不要直接 new 某个具体大模型。
所有 Provider 都要从 Factory 创建。


六、接入真实订单系统
--------------------

当前订单查询是 Mock：

BUSINESS_PROVIDER=mock

Mock 数据在：

app/providers/business.py

其中 MockBusinessProvider 里写了：

订单号 10001 -> 已发货，物流单号 SF10001


要接入真实订单系统，先改 .env：

BUSINESS_PROVIDER=http
EXTERNAL_INTERFACES_CONFIG=config/external_interfaces.yaml

然后修改 config/external_interfaces.yaml 里的：

business.order_query

举个例子，假设你的订单系统接口是：

GET https://orders.example.com/api/orders/10001

请求头：

Authorization: Bearer xxxxxx

返回：

{
  "id": "10001",
  "status": "shipped",
  "trackingNo": "SF10001"
}

那么 HttpBusinessProvider.get_order 最终要转换成项目统一格式：

{
  "order_id": "10001",
  "status": "已发货",
  "tracking_no": "SF10001"
}

为什么要转换？

因为项目里的 Action 不应该关心每个公司订单系统字段叫什么。
Action 只认统一字段：

- order_id
- status
- tracking_no

这样以后换订单系统，改 Provider 就行，不用改 Flow 和 Agent。

现在这个转换可以通过 response_mapping 配置完成。

如果真实接口返回：

{
  "data": {
    "id": "10001",
    "status": "shipped",
    "logistics": {
      "trackingNo": "SF10001"
    }
  }
}

那么 config/external_interfaces.yaml 可以这样写：

business:
  order_query:
    enabled: true
    url: "https://orders.example.com/api/orders/{order_id}"
    method: "GET"
    api_key: "你的订单系统密钥"
    timeout: 15
    headers:
      Authorization: "Bearer {api_key}"
      Content-Type: "application/json"
    body: {}
    response_mapping:
      order_id: "data.id"
      status: "data.status"
      tracking_no: "data.logistics.trackingNo"
    status_mapping:
      shipped: "已发货"
      paid: "已付款"
      pending: "待处理"
      cancelled: "已取消"

注意：

- {order_id} 会自动替换成用户输入的订单号。
- {api_key} 会自动替换成本段 api_key。
- data.logistics.trackingNo 表示从返回 JSON 里一层层取字段。


七、接入真实人工客服系统
------------------------

当前人工客服是 Mock：

HANDOFF_PROVIDER=mock

它会生成类似：

MOCK-AB12CD34

这不是真实工单。


要接入真实人工客服，改 .env：

HANDOFF_PROVIDER=webhook
EXTERNAL_INTERFACES_CONFIG=config/external_interfaces.yaml

然后修改 config/external_interfaces.yaml 里的：

handoff.create_ticket

假设你的客服系统接口是：

POST https://support.example.com/tickets

请求：

{
  "user_id": "user_001",
  "content": "我要转人工客服"
}

返回：

{
  "ticket_id": "T202506210001"
}

那么 create_ticket 要返回统一格式：

{
  "ticket_id": "T202506210001",
  "sender_id": "user_001",
  "message": "我要转人工客服"
}

现在可以这样配置：

handoff:
  create_ticket:
    enabled: true
    url: "https://support.example.com/tickets"
    method: "POST"
    api_key: "你的客服系统密钥"
    timeout: 15
    headers:
      Authorization: "Bearer {api_key}"
      Content-Type: "application/json"
    body:
      user_id: "{sender_id}"
      content: "{message}"
      source: "smart-customer-service"
    response_mapping:
      ticket_id: "ticket_id"

如果真实接口返回：

{
  "data": {
    "ticketId": "T202506210001"
  }
}

那么 response_mapping 要写：

response_mapping:
  ticket_id: "data.ticketId"


八、接入真实知识库
------------------

当前知识库来源是：

data/knowledge/return_policy.md

启动时，系统会读取这个 Markdown 文件。

当用户问：

你们支持七天无理由退货吗

系统会从这个示例文档里返回答案和来源。


如果你只是想改知识内容：

直接把 Markdown 文件放到：

data/knowledge/

然后重启服务。


如果你想做真实知识库，一般要做这些步骤：

1. 上传文档
2. 读取文档内容
3. 清洗文本
4. 切成小块 Chunk
5. 调用 Embedding 模型生成向量
6. 存入 pgvector
7. 用户提问时向量检索
8. Rerank 重排
9. 拼接引用来源
10. 交给大模型生成最终回答


当前项目里相关目录：

app/rag/document.py       文档读取、清洗、切分
app/rag/providers.py      Embedding 和 Rerank Provider
app/rag/vector_store.py   向量存储接口
app/rag/retriever.py      检索
app/rag/service.py        RAG 回答和引用格式


九、接入 BGE-M3 Embedding
-------------------------

BGE-M3 常用于中文知识库向量化。

.env 示例：

EMBEDDING_PROVIDER=bge-m3
EMBEDDING_MODEL=bge-m3
EMBEDDING_BASE_URL=http://你的embedding服务地址
EMBEDDING_API_KEY=你的密钥
EMBEDDING_DIMENSION=1024


如果你的 BGE-M3 是本地服务，需要确认：

- 服务地址
- 请求格式
- 返回向量字段
- 向量维度

然后新增真实 Embedding Provider，并在 app/providers/factory.py 中创建。


十、切换 pgvector
-----------------

当前默认：

VECTOR_STORE=memory

意思是知识向量只存在内存里，重启后就没了。


要切换 pgvector，改 .env：

VECTOR_STORE=pgvector
DATABASE_URL=postgresql+asyncpg://postgres:postgres@postgres:5432/customer_service


Docker Compose 里已经包含：

postgres:
  image: pgvector/pgvector:pg16


启动：

docker compose up --build


注意：

当前代码已经预留 VectorStore 接口，但 pgvector 的真实写入和检索还需要实现。
建议在 app/rag/vector_store.py 里新增 PgVectorStore。

十四、上线前必须检查
--------------------

上线前请检查：

1. .env 不要上传 GitHub。
2. LLM_API_KEY 不要写进代码。
3. BUSINESS_API_KEY 不要写进代码。
4. HANDOFF_API_KEY 不要写进代码。
5. DEBUG 生产环境建议改成 false。
6. 数据库密码不要用 postgres/postgres。
7. 真实 Provider 必须有 health_check。
8. 外部接口失败时必须有降级回复。
9. 订单、手机号、地址等敏感信息要按公司要求脱敏。
10. 日志不要打印完整 API Key。


十五、小白最容易混淆的点
------------------------

1. /docs 不是你的业务页面

/docs 是接口测试页面，不是给最终用户聊天的网页。
真正给用户用的前端页面以后可以单独开发。


2. /api/v1/chat 才是聊天入口

所有用户消息都应该发到 /api/v1/chat。


3. sender_id 很重要

同一个用户连续聊天，要用同一个 sender_id。
否则系统会以为这是另一个新用户。


4. Mock 不等于真实能力

Mock 只是为了让项目先跑起来。
真实上线必须替换 LLM、订单、人工客服、知识库等 Provider。


5. 改 .env 后要重启服务

很多配置只在启动时读取。
改完 .env 后，建议重启 uvicorn 或 docker compose。


十六、推荐接入顺序
------------------

建议不要一口气全换成真的。

推荐顺序：

第一步：保持 LLM_PROVIDER=mock，只接真实订单系统。

这样能先验证业务接口是否通。

第二步：接真实人工客服 Webhook。

第三步：把知识文档换成你自己的 Markdown。

第四步：接真实 Embedding 和 pgvector。

第五步：最后接真实大模型。

这样出问题时更容易定位。

十八、最小真实大模型接入清单
----------------------------

如果你只想先接真实大模型，需要完成：

1. 选择模型厂商，比如 DeepSeek、Qwen、OpenAI。
2. 获取 API Key。
3. 确认 base_url。

7. 确保大模型返回 CommandParser 能识别的 JSON。

CommandParser 期望类似：

{
  "type": "start_flow",
  "name": "query_order",
  "confidence": 0.95
}

或者：

{
  "type": "knowledge_qa",
  "confidence": 0.95
}


十九、最终判断是否完全跑通
--------------------------

完成真实接口接入后，至少跑这些命令：

python -m compileall app
uv run pytest -q
uv run ruff check .

然后启动：

uv run uvicorn app.main:app --reload

再测试：

1. /api/v1/health 正常。
2. /api/v1/chat 能正常聊天。
3. 查真实订单能返回真实物流。
4. 转人工能在真实客服系统生成工单。
5. 知识问答能返回真实知识来源。
6. 外部接口故障时不会让系统崩溃，而是返回友好的降级提示。

二十一、统一外部接口配置文件
----------------------------

现在项目已经新增一个统一模板：

config/external_interfaces.example.yaml

真实使用时，不要直接改 example 文件。
请复制一份：

cp config/external_interfaces.example.yaml config/external_interfaces.yaml

然后在 .env 里确认：

EXTERNAL_INTERFACES_CONFIG=config/external_interfaces.yaml

如果你要接真实大模型：

LLM_PROVIDER=http

然后修改 config/external_interfaces.yaml 里的 llm 段：

- enabled 改成 true
- url 写真实大模型地址
- api_key 写真实密钥
- model 写真实模型名
- headers 写真实请求头
- body 写真实请求体模板
- content_path 写返回内容所在字段

如果你要接真实订单系统：

BUSINESS_PROVIDER=http

然后修改 business.order_query：

- enabled 改成 true
- url 写订单查询接口
- method 写 GET 或 POST
- api_key 写订单系统密钥
- headers 写请求头
- body 写请求体
- response_mapping 写返回字段映射
- status_mapping 写状态值翻译

如果你要接真实人工客服：

HANDOFF_PROVIDER=webhook

然后修改 handoff.create_ticket：

- enabled 改成 true
- url 写创建工单接口
- method 一般写 POST
- api_key 写客服系统密钥
- body 写要提交给客服系统的数据
- response_mapping.ticket_id 写工单号字段路径

这样做的好处是：

1. 小白不用到处翻代码找接口地址。
2. URL、API Key、字段映射都集中在一个文件。
3. config/external_interfaces.yaml 已经被 .gitignore 忽略，不会上传 GitHub。
4. 代码仍然遵守“业务代码不能直接实例化具体 Provider”的要求。

注意：

不是所有公司的接口格式都一样。
如果你的真实接口非常特殊，比如需要签名、加密、分页、多次请求合并，
那仍然需要在 Provider 里补少量适配代码。
但普通的 URL、Header、JSON Body、JSON 字段映射，已经可以通过这个配置文件完成。
